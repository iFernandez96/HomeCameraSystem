# Mobile/Desktop/Orientation Coherence Review — 2026-07-07

**Summary:** The 2026-07-07 "landscape pass" (`e7e82e3` onward) did real, above-average work on three surfaces — Watch, Events, BottomNav/WatchRibbon — giving them a genuine `landscape-phone:` custom-variant treatment (`@custom-media (orientation: landscape) and (max-height: 520px)`, `index.css:16`). But it stopped there: **ClipModal, the single highest-stakes video surface in the app after Watch, got zero landscape-phone treatment**, and **Watch itself has zero `lg:` desktop treatment** — it ships the identical 48dvh-capped, `mx-4`-margined mobile stack to a 1440px monitor. Fix the ClipModal gap and give Watch a real desktop layout and you close both ends of the complaint in one pass; the rest is smaller inconsistency cleanup.

## Per-page verdict

### Watch (`pages/Watch.tsx`) — home route
- Portrait mobile: strong. Google-Home hybrid (video pinned top, timeline below) is the right call for a glance-check task.
- Landscape phone: strong — `landscape-phone:grid-cols-[58%_1fr]` two-pane reflow (`pages/Watch.tsx:302-343`) is exactly the right move, built and verified against a real SM-S928U1.
- Desktop (`lg:`): **absent**. `grep -c "lg:" pages/Watch.tsx` = 0 real hits (one is a comment). The video tile stays `aspect-video max-h-[48dvh] mx-4` (`Watch.tsx:343`) — a portrait-mobile-shaped card, centered in a sea of cream on a 1440px+ screen, with the timeline stacked below it in a single column. This is the file's own landscape-phone comment (`Watch.tsx:302-310`) literally naming the risk ("a short-but-wide lg window can render the WatchRibbon while this grid is active") without ever building the lg equivalent.
- Coherence score: **C** — best-in-class for the two phone orientations, an outright regression to "smaller desktop" for the one surface that most needs a real desktop layout (glance-check task is fine small; "look at live video AND today's story at once" is the desktop task, and nothing here serves it).

### Events (`pages/Events.tsx`)
- Mobile: single-column timeline, portal calendar overlay, sticky day-header — good density discipline.
- Desktop (`lg:`): genuinely two-pane — `lg:flex lg:items-start lg:gap-4` main + sticky 320px calendar rail (`Events.tsx:1333`), bulk-select action bar gated `selectionMode` for multi-select ops. This is the one page in the app that actually answers "what's the different desktop task" (bulk review vs. glance scroll).
- Landscape phone: has `landscape-phone:` overrides (`Events.tsx:954-955, 1333, 1383, 1399, 1572`) but they only re-center/pad — no reflow like Watch's grid. Acceptable since Events is naturally a scroll-down list either orientation.
- Coherence score: **B+** — the model other pages should copy. Docked wide but a real `xl:` opportunity is left on the table (see DESKTOP failures).

### Settings (`pages/Settings.tsx`)
- Mobile: horizontal scrollable tab strip (`flex gap-1 ... overflow-x-auto`, `Settings.tsx:329`), single-column sections.
- Desktop: `lg:flex lg:gap-6 lg:max-w-5xl` with a genuine `lg:sticky lg:top-20` vertical tab rail (`Settings.tsx:152,317-329`) — correct desktop pattern (persistent nav + content).
- Landscape phone: **zero `landscape-phone:` hits**. Harmless here — Settings is a scrolling form either way — but it's the only major page that didn't even get evaluated, so I can't tell you it was a deliberate "skip," it reads like an oversight.
- Coherence score: **B** — solid desktop split, unverified landscape.

### Login (`pages/Login.tsx`)
- Not read in depth this pass (App.tsx: `isLoginRoute` hides all shell chrome, single full-viewport brand block) — low risk, single-purpose screen, consistent with mobile-first auth conventions on both form factors. No finding.

## MOBILE failures

1. **`ClipModal.tsx:822,851,927` — landscape phone gets the portrait stack, not a reflow.** MOBILE (<640px, landscape, height<520): `flex flex-col lg:flex-row` collapses to `flex-col` because `lg:` doesn't fire below 1024px width — a rotated phone in landscape gets the **portrait** stacking (header → `aspect-video`-capped video → action row → the entire evidence `<aside>` scrolled below) inside a viewport that's often <400px tall. The video — already 16:9 — gets squeezed to a narrow width-driven strip instead of using the wide short viewport it's actually sitting in. DESKTOP (≥1024px): fine, `lg:flex-row` two-pane. WRONG on landscape mobile — this is the exact shape `landscape-phone:` was invented to fix (see `Watch.tsx` two-pane precedent), and ClipModal — arguably the #1 place a user rotates their phone to actually *watch* something — was skipped. FIX: give ClipModal the same `landscape-phone:grid-cols-[X%_1fr]` treatment Watch.tsx got: video pane docks left at full pane height, evidence aside becomes the independently-scrolling right column, matching (not duplicating) the existing `lg:flex-row` split so the same CSS shape now covers landscape-phone AND desktop instead of two different one-off rules. Effort: **M** (mirrors an existing pattern, needs real-device verification per CLAUDE.md convention). Iter: next in the landscape-pass thread (iter-356.6x).
   > Priya: "You built the two-pane reflow once, on Watch, and proved it works on a real Samsung. Then you didn't reuse it on the modal where people actually rotate their phone to watch the clip. That's not a design decision, that's fatigue."

2. **`pages/Settings.tsx` — no `landscape-phone:` audit at all.** MOBILE landscape: the vertical tab rail never activates (only `lg:` triggers it), so Settings falls back to the horizontal `overflow-x-auto` tab strip — which is *fine* — but nobody verified this, unlike Watch/Events/BottomNav which have explicit "real-device screenshot" comments. FIX: cheap — screenshot Settings at 926×428 (Galaxy landscape), confirm the sticky vertical rail doesn't half-render under `lg:`. If clean, add a one-line comment saying so (the codebase's own convention per CLAUDE.md's "landscape pass" commits) so the next person doesn't have to re-derive it. Effort: **XS**. Iter: landscape-pass sweep.

## DESKTOP failures

1. **`pages/Watch.tsx` — zero `lg:` classes, entire home route is a stretched mobile page on desktop.** DESKTOP (≥1024px, e.g. 1440×900): video tile stays capped at `max-h-[48dvh]` inside `mx-4` margins (`Watch.tsx:343`) — on a 1440px-wide window with SideRail (64px) and `lg:max-w-[calc(100%-4rem)]` (App.tsx:209), the video card sits centered with enormous horizontal dead space on both sides, exactly the "60% of horizontal space empty" smell from the audit brief. The glance row (`Watch.tsx:505`, `flex gap-2.5`) and today's timeline (`Watch.tsx:531-649`) stack below it in one column, unchanged from phone. MOBILE: correct — this IS the right glance-check shape at 390px. WRONG on desktop, where the task is "monitor live video AND scan today's activity simultaneously," which is a side-by-side task, not a stacked-scroll task — the exact two-column shape the `landscape-phone:` grid ALREADY proves the team knows how to build (`Watch.tsx:302`), just never extended to `lg:`. FIX: add an `lg:grid lg:grid-cols-[minmax(0,1fr)_380px] lg:items-start` (or reuse the landscape-phone grid verbatim at `lg:`) — video pinned left at a real desktop aspect (not capped to 48dvh, a mobile viewport-height hack that means nothing on a 900px-tall monitor), glance cards + timeline in a persistent right rail that scrolls independently, matching Events' rail pattern for visual family resemblance. Effort: **M**. Iter: pick up immediately after the ClipModal fix — same root cause (landscape-phone two-pane logic never made the jump to `lg:`).
   > Priya: "You have TWO successful desktop-density references in this same codebase — Events' calendar rail and Watch's own landscape-phone grid — and the app's actual home screen ships neither. On a 27-inch monitor 'On watch · 0 today' sits in a phone-width card with 900 idle pixels on each side. That's not responsive, that's centered."

2. **`components/EventList.tsx:180` — deliberately reverted to `lg:max-w-3xl` single column even inside Events' own two-pane desktop shell.** DESKTOP: `EventList`'s wrapper caps at `max-w-3xl` (768px) and centers itself INSIDE Events' `flex-1 min-w-0` column, which itself sits beside a 320px calendar rail inside a `lg:max-w-6xl` (1152px) container. Net effect on a 1440px+ window: timeline column has ~700-800px of real width available before the rail, but visually caps at 768px and centers — leaving asymmetric dead margins between the timeline and the calendar rail that read as a layout bug, not a design choice, even though the code comment (`EventList.tsx:159-179`) says it's intentional ("Maya verdict: three-column grid is a Stripe cliché, ship a timeline"). The REASONING (density-vs-legibility) is sound; the WIDTH MATH wasn't re-checked once the calendar rail was added alongside it later — the two decisions (kill the grid, add the rail) shipped in different iters and never got reconciled against each other's math. MOBILE: fine, single column is correct. FIX: either let the timeline column fill its actual available width up to the rail (drop the fixed `lg:max-w-3xl`, let row internals — not the wrapper — cap text-block legibility at ~65ch) or accept the current width and left-align instead of centering so the calendar rail reads as adjacent, not orphaned. Effort: **S**. Iter: whichever iter next touches EventList.
   > Priya: "'A timeline, not a grid' was the right call. Nobody went back and checked whether a 768px timeline centered next to a 320px rail inside a 1152px shell still reads like a timeline or like a mistake."

## COHERENCE failures

1. **Three different breakpoint conventions coexist in one component.** `components/WatchRibbon.tsx:130,145,170,186` mixes Tailwind's default `lg:` (brand wordmark, jump button, center alignment) with `md:` (last-frame-age visibility, lines 170/173) and the project's own custom `landscape-phone:` variant (line 108, margin offset) — three different mental models for "is there enough room" in one 190-line file. `md:` appears NOWHERE else in the five files this audit sampled (App.tsx, SideRail.tsx, BottomNav.tsx, Watch.tsx, Events.tsx all use only `lg:` + `landscape-phone:`), so this is a one-off drift, not a documented convention. FIX: replace the two `md:inline` hits with `lg:inline` (the last-frame age is exactly the kind of secondary info that should show at the same breakpoint the brand wordmark does, not one stop earlier) — one less threshold vocabulary for the next editor to hold in their head. Effort: **XS**. Iter: any touch of WatchRibbon.
   > Priya: "Three breakpoint systems in one 190-line header component. Pick one. `md:` isn't even used anywhere else in this app — it's a stowaway."

2. **`SideRail` (desktop-only, `lg:flex`) and `BottomNav` (mobile-only, `lg:hidden`, landscape-dock via `landscape-phone:`) share near-identical `NAV_ITEMS`/`tabs` arrays but SideRail has 5 items, BottomNav has 4 visible + 1 `landscapeOnly`.** This is actually a GOOD divergence (documented at `BottomNav.tsx:18-27` — "portrait pebble bar deliberately stays at 4... landscape-phone dock has the same vertical room as desktop rail") but it means there are now THREE nav item counts to keep in sync across TWO files with hand-copied SVG glyph functions (`LiveIcon`, `EventsIcon`, etc. are byte-identical, duplicated in both files with "keep in sync" comments — `BottomNav.tsx:148-149,165-166,186-187`). That's a maintenance coherence risk, not a UX one: the icons WILL drift the next time someone edits one file under time pressure. FIX: extract the shared glyph set into `components/NavIcons.tsx`, both SideRail and BottomNav import from it; keep the per-surface item-count/label logic local. Effort: **S**. Iter: any nav touch.

## COMPONENT split candidates

- **`Watch.tsx`'s docked video card** — currently one JSX block wearing three hats via `full ? ... : ...` ternaries plus bare `landscape-phone:` overrides layered on top (`Watch.tsx:320-344`). It's manageable today but every future orientation/breakpoint addition adds another ternary branch to the same className string. Once desktop `lg:` support is added (see DESKTOP finding #1), this is the point to extract a `<LiveViewportFrame density="mobile-glance" | "desktop-rail" | "landscape-dock" | "fullscreen">` wrapper that owns the box-model decision, so VideoTile itself stays orientation-agnostic (it already is — zero `lg:`/`landscape-phone:` in VideoTile.tsx, which is correct, that component should never know about page layout). Effort: **M**, bundle with the desktop-Watch fix above rather than doing it standalone.
- NOT `ClipModal` — its `lg:flex-row` split is a clean, single-condition branch; extracting two components there would be over-engineering for one boolean. Just extend the existing conditional with `landscape-phone:` per the MOBILE finding above.

## TASK divergence opportunities

- **Watch home**: mobile task is "glance and confirm the house is fine" (served well). Desktop task should be "keep the feed up while working nearby, scan for anything from today without leaving the tab" — that wants the video SMALLER (not `max-h-[48dvh]`, more like a fixed ~320px rail width) and the timeline TALLER/wider, closer to Slack's persistent-video-call-tile-plus-channel-list shape than Google Home's tap-to-focus card. Currently ships identical proportions to mobile. Directly caused by DESKTOP finding #1.
- **Events review**: already diverges well (bulk-select desktop-only, calendar overlay-vs-rail) — good precedent, no further ask.
- **ClipModal**: mobile task is "confirm what triggered the alert, maybe delete it" (full-page focus, correct). Desktop task should add keyboard shortcuts (arrow-key next/prev clip, `Del` to delete) since a desktop user reviewing many clips in a sitting is exactly the precision/multi-item workflow `lg:flex-row` split-pane implies but doesn't yet exploit — currently every action (`Delete`, `Save`) is mouse/tap-only. Not scored as an A11Y finding because nothing is BROKEN, but it's a missed desktop-specific affordance. Effort: **M** if picked up, own iter.

## A11Y per-form-factor

- No new regressions found in the five sampled files beyond what CLAUDE.md already documents as fixed (SideRail/BottomNav `aria-hidden` parity, WatchRibbon's `role="status"` scoping to avoid double-announce, skip-link). The nav SVG dual-file duplication (COHERENCE #2) is the closest thing to an a11y risk — if the two files drift, a future `aria-label` fix applied to one glyph set and not the other would silently reintroduce a screen-reader mismatch.

## Anti-recommendations

Things deliberately NOT flagged, because the form-factor cost is low or the existing choice is already correct:

1. **`People.tsx`/`Training.tsx` sharing `max-w-3xl lg:max-w-4xl` + `lg:grid-cols-2/3/4` grids (iter-342 desktop A1).** These are genuinely dense multi-select-shaped grids on desktop and single-column lists on mobile — the right split, not flagged.
2. **`VideoTile.tsx` having zero `lg:`/`landscape-phone:` classes.** This is correct, not a gap — VideoTile is a pure video-rendering primitive; every page that embeds it (Watch, ClipModal) owns the box it sits in. Keeping VideoTile orientation-agnostic is the right architecture and should stay that way even after the Watch/ClipModal fixes above.
3. **`EventHeatmap.tsx` having zero `lg:` classes.** Its `grid-cols-7` fluid layout already scales correctly from a 320px portrait sheet to a 320px desktop rail (Events.tsx wraps it, not the heatmap itself) — no breakpoint logic needed inside the component, matching the loop_audit_log 5289 note ("mobile default closed, cost is opt-in").
4. **`Settings.tsx` skipping `landscape-phone:` entirely** given the horizontal tab strip already degrades acceptably (see MOBILE finding #2 for the XS-effort verification ask — flagged as "unverified," not "broken").
