# face_recog

Per-person identification on top of the object detector. When the worker
sees a `person` detection, it runs face recognition against a small
known-faces database and emits the recognized name as `person_name` in
the event payload (falling back to null when nothing matches above the
tolerance — the event still publishes with the generic `person` label).

Why the dir is named `face_recog` and not `face_recognition`: the pip
package we depend on is also called `face_recognition`, and Python's
import system can't tell them apart if the local dir shadows the package
name. Keeping the dir name distinct lets the wrapper `import
face_recognition` cleanly.

## Files

```
refs/photo_NN.jpg        14 reference shots (Israel + Sheenal, mostly together)
manifest.json            per-photo face labels, left-to-right
encode_known_faces.py    builds encodings.pkl from refs + manifest
recognizer.py            FaceRecognizer class used by detect.py at runtime
encodings.pkl            generated by the encoder; not committed
```

## Install on the Jetson — current status: BLOCKED

`face_recognition` wraps `dlib`. **dlib v20.0 currently deadlocks at
import on the Jetson Nano 2GB regardless of CUDA**; the worker boots
fine without it (see "Worker integration" below — `init_face_recognizer`
returns None gracefully when `encodings.pkl` is missing) but the
encoder + recognizer can't actually do anything until dlib is unblocked.

Two distinct hangs we've diagnosed:

1. **CUDA-linked wheel** (the one pip3 grabs by default): static init
   tries to spin up a CUDA context inside dlib and never returns on the
   resource-starved Nano 2GB.
2. **CUDA-disabled rebuild** (`python3 setup.py install --user --no
   DLIB_USE_CUDA` from a v20.0 source clone): different deadlock — gdb
   shows `__pthread_once_slow` calling an init routine that tries to
   `PyEval_AcquireThread` while the same thread already holds the GIL.
   Recursive cond-wait, never returns.

Both are documented in the project's memory file `jetson_dlib_no_cuda.md`.

**Unblock paths to try in a future iter** (ordered by best guess at
effort):

1. Pin to a pre-2024 dlib (v19.18 / v19.21 / v19.24) — known-stable on
   Jetson Nano 2GB across multiple community projects, presumably
   without the v20.0 GIL bug.
2. Switch backend entirely to InsightFace + onnxruntime. The
   `FaceRecognizer.recognize_in_crop()` wrapper isolates the backend
   so the rest of the stack (detect.py integration, server schema,
   client UI) is unaffected.
3. Patch dlib v20.0's static init to drop the `pthread_once`
   GIL-acquire pattern. Smallest scope but requires a debugger session
   on the Jetson.

If/when one of those works, `face_recognition` install on the Jetson
is just `pip3 install --user numpy face_recognition` (the Python
wrapper itself is fine; only dlib is broken).

## Build encodings

After install, run the encoder on the Jetson against a synced copy of
this directory:

```bash
ssh jetson 'cd ~/HomeCameraSystem/detection/face_recog && python3 encode_known_faces.py'
```

The script prints a count per person, e.g.:

```
[encode] wrote encodings.pkl (24 encoding(s))
[encode]   israel: 12
[encode]   sheenal: 12
```

## Manifest

`manifest.json` maps photo file → list of person names, in left-to-right
order of the detected faces. The current values are best-guess labels
from inspecting all 14 photos. **Verify them on the Jetson** — if a face
is mislabeled, fix the entry, re-run the encoder. Special tokens:

- `"_skip"` — skip this face (use for low-quality faces, partial faces,
  or unknown people in group shots like the rollercoaster photo).
- `"unknown"` — same as `_skip` (any non-target name is dropped).

## Worker integration

`detection/detect.py` loads the encodings on startup and, after the
cooldown gate fires for a `person` detection, crops the upper portion of
the bbox and calls `FaceRecognizer.recognize_in_crop()`. The matched
name (or null on no match) is sent as `person_name` in the event
payload.

Threshold tuning lives in `recognizer.py::FaceRecognizer.tolerance`
(default 0.55 — slightly stricter than face_recognition's default 0.6 to
reduce false positives on a small library).

If `encodings.pkl` is missing or unreadable, the recognizer logs a
warning and degrades gracefully — events still flow with the generic
`person` label and `person_name: null`.

## Python 3.6 compatibility

`encode_known_faces.py` and `recognizer.py` both run on the Jetson
host where JetPack 4.x ships Python 3.6 — same constraint that the
top-level CLAUDE.md sharp edges already pin for `detection/*.py`.
**Don't add** `from __future__ import annotations`, PEP-604 unions
(`int | None`), `match` statements, or walrus (`:=`) here. The
encoder is invoked directly via `python3 encode_known_faces.py` and
the recognizer is imported by `detect.py` at worker boot — either
crashes the homecam-detect systemd unit at import time if 3.7+
syntax sneaks in (iter-28 burned ~42 systemd restarts on this
exact bug before it was caught).

## Why dlib and not "Meta's face DB"

Meta shut down their facial-recognition system and deleted the
underlying face vectors in 2021 (privacy class-action settlement) — no
public DB exists.

The dlib model bundled with `face_recognition` is a ResNet-34 pre-trained
on millions of public faces (FaceScrub + VGG-Face) and scores ~99.38 %
on the LFW benchmark. That's already in the same accuracy class as the
2014 DeepFace paper. The reference photos in `refs/` aren't training
data — they just give the model 128-d anchor vectors for the people we
want to identify, which is the only piece a more famous dataset
wouldn't help with.

Higher-accuracy alternatives (InsightFace/ArcFace ONNX, FaceNet TFLite)
exist but trade ~200 MB extra weights + a separate face-detector
pipeline for sub-percentage-point gains that don't matter when there
are only two people to distinguish.
